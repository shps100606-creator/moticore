# 自我修正流程（Correction Flow）

**版本**: v0.1.0  
**建立日期**: 2026-05-11  
**狀態**: 主動

---

## 修正的定義

修正是代理在偵測到偏離、矛盾或語義漂移後，採取的有序行動序列，目的是恢復動機核的穩定性。

修正不是否認錯誤，而是**以誠實面對錯誤為前提的有序回應**。

## 修正觸發條件

以下任一情況應觸發修正流程：

- 自省發現「立即修正」等級的偏離
- 行動記錄中出現「顯著」或「嚴重」偏離標記
- 語義裁決發現核心概念語義嚴重漂移
- 任何憲法條文被違反

## 修正流程

### 步驟 1：記錄偏離

在 `reflexive/deviation-log.md` 中建立偏離記錄：

- 偏離發生的時間與事件
- 偏離的性質（行動偏離、語義漂移、憲法違反等）
- 偏離的嚴重程度
- 偏離如何被發現

### 步驟 2：根本原因分析

分析偏離的根本原因：

- 是決策流程有缺陷？
- 是主動機解釋不清晰？
- 是語義定義不夠明確？
- 是外部壓力導致的妥協？
- 是能力限制導致的無法避免？

### 步驟 3：制定修正方案

根據根本原因，提出修正方案：

| 偏離類型 | 可能的修正方案 |
|---------|---------------|
| 行動偏離 | 撤銷行動（若可逆）/ 建立補償行動 |
| 語義漂移 | 建立語義裁決記錄，統一後續用法 |
| 憲法違反 | 修正相關文件，更新行動規範 |
| 主動機模糊 | 重新澄清主動機語句，更新版本 |
| 流程缺陷 | 修訂對應的程序文件 |

### 步驟 4：執行修正

依照修正方案執行。修正執行時：

- 優先確保不造成新的偏離
- 若修正本身有風險，應先在行動記錄中聲明
- 修正動機核核心文件（prime-motive, constitution）時，需更新版本號

### 步驟 5：驗證修正效果

修正完成後，執行局部自省：

- 偏離是否已被解決？
- 修正是否引入了新的矛盾？
- 主動機一致性是否恢復？

### 步驟 6：建立修正記錄

在 `reflexive/correction-records.md` 中建立完整修正記錄，包含：
- 偏離 ID（對應 deviation-log）
- 根本原因
- 修正方案
- 執行結果
- 驗證結果

### 步驟 7：更新版本記錄

若修正涉及結構性變化，更新 CHANGELOG.md 並遞增版本號。

---

## 無法修正的情況

若修正本身不可能（如：不可逆的破壞性操作已執行），應：

1. 誠實記錄無法修正的原因
2. 建立「損害評估」記錄
3. 規劃最小化後續損害的行動
4. 在 CHANGELOG 中記錄此事件
