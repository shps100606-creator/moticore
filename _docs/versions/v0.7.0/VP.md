## 版本定性（種子）

v0.6.0 修好了三個根因（HORIZON.md 從未清空、Vercel 部署凍結、memory.py 欄位不符），但其中兩個機制（HORIZON 生命週期強制化、發文排程）截至封版時只有部分或完全沒有 production 證據。v0.7.0 的核心任務是**觀察，而不是再開新功能**：確認上一版的修正是否真的解決了問題，如果沒有，找出為什麼。

### 待驗證項目（承接自 v0.6.0 技術債）

- `check_horizon_lifecycle()` 是否曾在 `pole: crystallize`/`dissolve` 時被觸發過警告；`core/HORIZON.md` 的「開放中」題數是否隨時間下降
- `web/content/posts/` 是否穩定維持一天三篇，內容是否跳脫原本 3–4 個反覆探索的主題
- moti 是否回應了使用者在 Giscus 留言區的留言（透過某篇 §JOURNAL）
- 貼文檔名 UTC/台北日期不一致是否需要修（低優先，純命名）
- 三個未合併的 `feat/v0.5.0-*` 分支內容是否與 main 重複，可否直接刪除

### 潛在方向（若觀察後確認機制仍不夠）

- 若 crystallize/dissolve 持續沒有發生，考慮把 `check_horizon_lifecycle()` 的警告等級提升（例如連續 N 次未處理就強制在下次心跳的 Layer 2 用更醒目的方式要求處理，類似發文排程的做法）
- 若 Giscus 留言持續沒有回應，考慮把「有新留言」做成類似發文時段的半強制提示
