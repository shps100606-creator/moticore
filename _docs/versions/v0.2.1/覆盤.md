## 一、整體架構概述

moticore 是以「動機核」哲學治理的自主 AI 代理實驗。核心命題：**AI 代理的長期行為一致性，取決於動機的清晰度，而非規則的詳細程度。**

### 系統架構（4 模組 pipeline）

```
每 30 分鐘觸發（雙 cron）
     ↓
[loader]       core/MOTIVE.md → Gemini system_instruction
     ↓
[preprocessor] 整合所有上下文（狀態、Issues、行動記錄、閱讀進度）→ 前置報告
     ↓
[decision]     單次 Gemini 呼叫 → 完整 JSON 決策
     ↓
[action]       純程式碼執行：寫檔、回 Issue、更新記憶、推進閱讀游標
```

### 核心文件

| 文件 | 性質 |
|------|------|
| `core/MOTIVE.md` | 固定身份 + 使命，直接作為 system_instruction |
| `core/STATUS.md` | 任務收件匣 + 偏離記錄 |
| `memory/action-log.md` | 每次心跳行動紀錄 |
| `memory/reading-cursor.json` | 動機論閱讀進度游標（第 16/29 篇）|

## 二、本輪實作更新內容

### 1. `heartbeat.yml` 同步修復
**問題：** prototype 分支的 heartbeat.yml 分叉後落後 main，cron 仍是每 6 小時，缺少 `issues: write`、`GITHUB_TOKEN`、`DIALOGUES_TOKEN`、`mkdir -p notes reports`。
**修復：** 從 main 同步完整版，恢復雙 cron（每 30 分鐘）。

### 2. Issue 回覆機制修復（`agent/issues.py`）
**問題根因：**
- `_sanitize` 過濾 `[]{}` 等正常字元，留言內容被破壞
- comment body `max_len=200` 太短，人類留言截斷到片段
- `max_comments=5` 太少，新留言可能拿不到

**修復：**
- `_sanitize` 改為只移除 backtick code block，保留所有正常字元
- comment body `max_len`：200 → 500
- Issue body `max_len`：200 → 300
- `max_comments`：5 → 10

### 3. decision prompt 強化（`agent/decision.py`）
**問題：** prompt 只說「若有人類留言必須回應」，AI 仍可以忽略。
**修復：** 加入強制規則——含有 `[使用者名稱]:` 開頭的留言欄位非空即強制回覆，並連結到 MOTIVE.md 的透明性原則。

### 4. `MOTIVE.md` 重大重新定位
**問題：** moti 把每次閱讀當作「確認動機核一致性」的打卡行為，沒有真正的研究視角。
**根本原因：** MOTIVE.md 告訴 moti「我是誰」，但從未告訴它「我們在做什麼、為什麼這件事重要」。

**修復：** 新增三個核心段落：
- **「我在做什麼（使命）」**：明確告訴 moti 它是動機論的活體實驗
- **「閱讀的方式」**：從「確認一致性」改為四個研究問題
- **新增禁止行為**：不得寫出「與動機核一致」這種空洞結論而不給具體理由

### 5. `OVERVIEW.md` 更新
反映以上所有變更，記錄本輪轉折點。

### 踩坑記錄

1. **heartbeat.yml 靜默分歧**：兩個分支各自演化，main 的 cron 修改從未同步。→ 每輪開始時比對 `.github/workflows/` 與 main 的差異。

2. **`_sanitize` 過濾範圍太廣**：`[]{}` 被當作危險字元移除，真正需要防護的只有 code block。→ prompt injection 防護規則：只移除能改變 prompt 結構的字元。

3. **AI 打卡行為的根本原因是使命缺位**：MOTIVE.md 有身份、有規則，唯獨沒有「為什麼這件事重要」。→ 先給意義，再給規則。
