## 主題

使用者提供一張 GitHub Actions `heartbeat` 執行失敗的截圖（`Commit changes` 步驟 `git push` 被拒），本 session 追查根因、確認實際發生頻率，並依使用者當場決定暫停 moticore 的自動運作。

## 內容 / 關鍵結論

1. **根因確認**：`.github/workflows/heartbeat.yml` 的「Commit changes」步驟只有 `git add` → `git commit` → `git push origin main`，中間沒有 `git pull`/rebase 或重試機制。雙 cron（`0 * * * *` + `30 * * * *`）在 GitHub Actions 排程延遲追趕時，偶爾兩個 runner 幾乎同時觸發，各自從同一份 `main` checkout 後各自寫入，先推的成功、慢一步的被 `! [rejected] main -> main (fetch first)` 拒絕，整個 job 失敗（exit 1），**該次心跳的全部本地寫入被整個丟棄**。

2. **實際頻率**：查了 2026-07-18T20:51～07-21T07:04 約 60 次執行，有 2 次遇到一模一樣的錯誤（07-19 23:48 run #1339、07-20 02:06 run #1341，兩次間隔約 2.5 小時），log 內容完全一致。頻率不算高（約 3%），但每次代表一次心跳完全失去記錄。

3. **與 WN1 D2（Giscus 回覆機制）的潛在交互風險**：`agent/run.py` 呼叫 GraphQL `addDiscussionComment` 是外部 API 動作，一旦執行即已真的發到 Giscus，不會因後續 git push 失敗而收回；但用來防止重複回覆的 `memory/giscus-replied.json` 是本地檔案，若剛好在寫入後 push 失敗，下次心跳會讀到舊版、可能對同一則留言重複回覆。截至目前尚未實際觀察到（07-09 後也沒有新留言可測），但這是一個真實存在、先前未被記錄過的技術債。

4. **使用者當場決定**：在看到這個根因分析後，使用者表示「這個專案運作到現在，幾乎沒有產生任何意義」，決定：
   - 先把雙 cron 合併為單一 cron（`*/30 * * * *`，效果與原本相同，但消除兩條規則各自延遲追趕、重疊觸發的模糊性），修正上述 bug 讓程式碼留在乾淨狀態
   - 隨即**暫停**專案的自動排程（`.github/workflows/heartbeat.yml` 的 `schedule:` 觸發器整段註解停用，只保留 `workflow_dispatch` 供未來手動恢復），不再持續產生心跳

5. **對照 WORK_260709_002（waiting：觀察 dissolve 極是否觸發、Giscus 機制長期穩定性）**：截至暫停當下，dissolve 極累計仍是 0 次觸發（`core/HORIZON.md`「開放中」四題全部只有 crystallize 方向的探索進展），Giscus 機制因缺乏新留言仍未被長期驗證。這兩項觀察指標最終沒有等到明確結論，專案就因使用者對整體產出價值的判斷而暫停，並非因為技術指標本身給出「可以封版」或「不該封版」的訊號。

## 未完成任務
- 若未來要恢復，需先把 `.github/workflows/heartbeat.yml` 的 `schedule:` 取消註解（已是合併後的單一 cron），並重新評估是否要在 push 前加 `git pull --rebase` 之類的保護，徹底根除本筆記記錄的 race
- WORK_260709_002 的兩個觀察指標（dissolve 極、Giscus 長期穩定性）在暫停後失去繼續累積資料的管道，若未來恢復需重新起算觀察期

## 未解決問題
- dissolve 極从 v0.7.0 起持續 0 次觸發，到暫停為止都沒有解釋——是機制尚未遇到觸發條件，還是「顯著偏離」抓包對「不宣稱」情境本來就無約束力（v0.7.0/v0.8.0 已反覆指出的已知邊界），留待未來恢復專案時再議
