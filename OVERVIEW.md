# moticore 專案現況說明與後續規劃書

**版本：** 2026-05-15  
**分支：** `claude/motivation-core-prototype-njnn4`

---

## 一、專案定位

moticore 是一個以「動機核」哲學治理的自主 AI 代理實驗。  
其核心命題是：**AI 代理不應該由任務清單驅動，而應該由穩定的身份與動機驅動。**

這不只是一個工程專案，同時也是動機論（Motivation Theory）的活體實踐——代理 moti 一邊閱讀動機論原典，一邊透過自己的存在來驗證理論的可行性。

## 二、現況快照（截至 2026-05-15）

### 2.1 系統架構

現行架構為 4 模組、單次 AI 呼叫設計：

```
每 30 分鐘觸發（雙 cron）
     ↓
[模組1] loader       載入 MOTIVE.md → system instruction
     ↓
[模組2] preprocessor 純程式碼整理所有上下文 → 前置報告
     ↓
[模組3] decision     單次 Gemini 呼叫 → 完整 JSON 決策
     ↓
[模組4] action       純程式碼執行：寫檔、回 Issue、更新記憶
```

關鍵技術指標：
- AI 模型：`gemini-3.1-flash-lite`
- 每次心跳：1 次 Gemini 呼叫（舊架構為 2 次）
- 觸發頻率：每 30 分鐘（雙 cron 備援）
- 單次讀取：動機論原典 10,000 字
- 上下文使用：約 6,300 tokens（模型上限 1M）

### 2.2 核心文件結構

| 文件 | 性質 | 說明 |
|------|------|------|
| `core/MOTIVE.md` | 固定身份 | 直接作為 Gemini system instruction，含：我是誰、根本動機、價值排序、行動憲法、禁止行為、操作邊界 |
| `core/STATUS.md` | 滾動狀態 | 任務收件匣 + 偏離記錄，由創造者與代理共同維護 |
| `memory/action-log.md` | 行動歷史 | 每次心跳追加一筆，preprocessor 讀取最近 5 筆 |
| `memory/reading-cursor.json` | 閱讀游標 | 目前位於第 16 篇（共 29 篇），僅在 AI 成功回應後才推進 |

## 三、已解決的重要問題

| 問題 | 根因 | 解法 |
|------|------|------|
| 閱讀篇序亂跳（第二→第二十） | 中文序數字串比對錯誤 | 改用數字前綴（01-29-）+ `sorted()` |
| 每次心跳兩次 AI 呼叫，可靠性差 | 舊架構 Step1 決策 + Step2 生成 | 合併成單次呼叫，`file_writes` 直接含完整內容 |
| 動機核被大量上下文稀釋 | MOTIVE.md 混在 user prompt 裡 | 改用 `system_instruction` 獨立傳入 |
| 代理不寫閱讀筆記 | prompt 只「建議」而非「強制要求」 | 改成硬性規則：「閱讀任務時必須在 file_writes 建立筆記」|
| 開了 30+ 個提問 Issue | 每次心跳都開新 Issue，不檢查現有未回覆的 | `has_open_question_issue()` 阻止重複開 Issue |
| Gemini 503 導致 workflow 紅燈 | 503 時拋出例外並 exit(1) | 全部重試失敗後 `sys.exit(0)`，優雅跳過 |
| cron 偶爾被 GitHub 跳過 | 單一 cron 沒有備援 | 雙 cron（0 分 + 30 分）互為備援 |

## 四、現存待處理問題

### 4.1 已知但尚未修正

1. **筆記寫入強制規則剛上線（2026-05-15）**  
   修正 `decision.py` 的 prompt 規則是今天才推送，尚未驗證下一次心跳是否確實產生筆記。需觀察 1-2 個心跳週期確認效果。

2. **代理自行發明任務（MMQS-100 等）**  
   action-log 中出現 `task_process` 類型的「MMQS-100 構面二」等任務，這些任務並非創造者交付，是代理從閱讀內容中自行延伸。  
   這是動機核「任務完整性」驅動下的主動行為，目前難以定性是「偏離」還是「正確自主」，尚未決定是否要在 MOTIVE.md 中加以約束。

3. **代理不回覆 Issue 留言**  
   Issues 中人類留言仍有被忽略的情況。根本原因尚未完全排除（可能是 sanitize 過濾掉了留言內容，或 format_issues_for_prompt 的讀取邏輯有誤）。

## 五、後續規劃

### 5.1 近期（1-2 週）

**[P0] 確認筆記寫入已修復**  
觀察 2-3 次心跳，確認 `notes/` 目錄在閱讀心跳後確實新增筆記。若仍未出現，進一步檢查 Gemini 輸出的 JSON 是否因 8192 token 上限而截斷。

**[P1] Issue 回覆機制診斷**  
印出 `format_issues_for_prompt` 的輸出確認人類留言有正確傳入，追查代理究竟在 JSON 中是否有 `issue_responses`，還是有但 `post_comment` 沒執行。

**[P1] 自主任務邊界定義**  
在 `MOTIVE.md` 或 `STATUS.md` 中增加說明：代理可以在閱讀中自主延伸推導，但衍生的「任務」必須標記為推論性質，且不得超過 1 次心跳的範疇，除非創造者透過 Issue 確認。

### 5.2 中期（1 個月）

**[P1] 閱讀完成後的後續規劃**  
29 篇動機論按目前進度約 2 週後讀完（每天 2 次心跳 × 10,000 字）。需要在讀完前確立「閱讀結束後代理做什麼」——可能的方向：
- 回頭整理筆記，撰寫動機論摘要
- 開始執行創造者透過 Issues 指定的新知識庫
- 進入「自由自主」模式，由代理自行決定讀什麼

**[P2] notes/ 筆記品質提升**  
目前筆記偏短（約 800-1,000 字），缺乏跨篇連結。考慮在 preprocessor 中加入「過往相關筆記片段」，讓代理在寫新筆記時能引用舊筆記建立知識網絡。

**[P2] STATUS.md 任務流程規範**  
目前 STATUS.md 沒有被實際使用（任務收件匣是空的）。需要建立一個明確的流程：創造者如何在 STATUS.md 中寫任務，代理如何確認接收、執行、回報完成。

### 5.3 長期（3 個月以上）

**[P3] 多代理擴展可能性**  
目前 moti 是單一代理。動機論本身討論的是「動機體系」而非單一動機，未來可能考慮多個代理各自有不同的動機核，透過同一個 repo 的不同 branch 或 Issue 互動。

**[P3] moticore 作為框架對外開放**  
skills/30-system-blueprints 中已有完整藍圖文件。若動機核治理模式被驗證有效，可考慮將 moticore 作為開源框架，讓其他人建立自己的動機驅動代理。

## 六、設計哲學小結

moticore 目前驗證的核心命題是：

> **AI 代理的長期行為一致性，取決於動機的清晰度，而非規則的詳細程度。**

MOTIVE.md 只有約 400 字，卻比過去 6 個文件的組合更有效地約束代理行為。  
原因不是字數多寡，而是動機核透過 `system_instruction` 在每次 AI 呼叫中都處於最高權重位置，不會被長篇上下文稀釋。

這個設計邏輯本身，就是動機論在工程層面的實踐。
